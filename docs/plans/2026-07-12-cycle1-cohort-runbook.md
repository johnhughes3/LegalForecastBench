## 0. Pre-Batch-001 gate: settle the packet-completeness definition

The first draft of this runbook defined a "clean" case as complaint + MTD memorandum + opposition + decision, with only reply optional. **The code disagrees**: `legalforecast/ingestion/public_packet_planner.py` sets `_OPTIONAL_BRIEF_ROLES = {OPPOSITION, REPLY}` — only complaint, target MTD document, and decision are core, and `tests/test_public_packet_planner.py:96-128` explicitly accepts the no-opposition case. This is not a nit: the definition drives N₀, N₁(k), projected purchase spend, and the benchmark construct itself. It must be decided and encoded **before Batch 001 measures anything**, or the measurement answers the wrong question.

Recommended resolution (decision bead, John approves): **completeness = every core document that was actually filed is present**, i.e.:

- Complaint, target MTD document, decision: unconditionally core (as the code has it). **Second correction from review: "target MTD memorandum" misstated even the current code** — the planner accepts an `MTD_NOTICE` when no free memorandum exists and records an MTD gap only when neither is available (`public_packet_planner.py:323-391,704-709,739-748`). A notice-of-motion without the supporting memorandum is *not* the argument the judge read, so the construct decision must settle this too. Recommended: **memorandum-or-combined-motion required; a bare notice is a gap** (same "record the judge saw" logic as opposition).
- Opposition: **conditionally core** — required when the docket shows an opposition was filed; not required when the motion was genuinely unopposed. An unopposed MTD decided on the papers is a legitimate, complete-record case; unconditionally requiring opposition (first-draft position) would wrongly exclude it. But the current code's unconditional "optional" is also wrong for the construct: a docketed-but-unavailable opposition means the model is forecasting from *less than the record the judge saw*, which silently changes what the benchmark measures and correlates missingness with RECAP-upload patterns. The decision must also pin down **which opposition(s)**: recommended = the opposition(s) linked to the *selected target motion* (not every opposition/joinder/corrected filing on the docket), which requires making that linkage explicit — see below.
- Reply: optional (both draft and code agree).

**Implementation scope, corrected (review point accepted — rev 2 called this "a small change to `public_packet_planner.py`"; it is not):** the change touches at least five surfaces, and an executor sizing it as a planner one-liner will break the bridge. (i) Planner: optional opposition entries without free documents are currently *discarded*, not gap-recorded (`public_packet_planner.py:789-807`), and opposition entries are not linked to the selected target motion (`:371-380,517-526`) — both change. (ii) Paid-gap reason vocabulary: a new `no_free_opposition` (and `no_free_mtd_memorandum` if the notice rule above is adopted) must be added to the bridge's `_PAID_GAP_ROLES` map (`courtlistener_case_dev_bridge.py:55-63`) or the bridge's `_PAID_GAP_ROLES[reason]` lookup raises on the new reason. (iii) Core-document filter and (iv) budget planner: the new roles flow into `filter-core-documents` and `missing_core_budget` so purchases and completeness counts see them. (v) Tests across all of the above, including the unopposed-motion and multiple-opposition fixtures. Medium bead, not small. Batch 001 reports yield under **both** definitions (unconditional-optional vs conditional-core) — computed **offline from the same frozen docket snapshots** (an analysis script over the planner inputs), not by maintaining two live planner configurations through the pipeline; after the decision, exactly one completeness semantics exists in production (review point (d)5 accepted).

## 1. The honest yield picture

The 150-clean-motion target assumes purchase capability. The free-only picture is much tighter, and the plan must be built around measuring it fast rather than assuming it away.

**Why free-only complete-briefing yield is small.** A "clean" case needs the complete pre-decision briefing set (per the §0 definition) freely on RECAP. Decisions are cheap to find free (written opinions are fee-exempt on PACER and CL's fetchers pull many), but *briefing* is free only where some RECAP-extension user actually paid for and uploaded each document. That coverage is (a) sparse, (b) concentrated in high-attention districts (SDNY, N.D. Cal., D. Del., E.D. Tex., D.D.C. and similar), and (c) biased toward newsworthy cases — which is a case-mix skew the diagnostics must report, not hide. The pre-plan survey estimated free-only complete yield at "a few tens" of cases for a fresh window; treat that as the prior, not the answer.

**Structural yield taxes already in the pipeline** (each shows up in the exclusion ledger, so Batch 001 quantifies them for free):

- Multi-page dockets are excluded outright (`courtlistener_docket_more_than_one_page`, `mtd_acquisition_screen.py:356`) — long dockets are disproportionately the big, well-uploaded cases, so this tax hits the free-rich stratum hardest.
- Unparseable disposition dates, non-civil/metadata screen failures, pre-anchor first dispositions, and strict-clean screen failures all fail closed (`courtlistener_acquisition.py::_screen_candidate`).
- The anchor itself: only decisions entered on/after 2026-06-30. As of today that is an 11-day window. **Time is a first-class lever: the cohort grows every week the collection window stays open.** Rough sizing: federal district courts resolve on the order of a few hundred Rule 12(b)(6)-family motions per week nationwide; RECAP-visible, single-page, strict-clean, complete-free-briefing survivors will be a low single-digit percentage of that. Do not commit to a final N before Batch 001 measures the funnel.

## 3. Purchase-vs-reduced-packet policy

Configured guardrails (`missing_core_budget.py:14-16`): $3.05/doc estimated, **$2,250 projected cap** (~737 documents), ≤24 missing-core docs/case. Note the cap is *projected*-spend only today; the keyless-run plan §4e purchase-safety design (rev 4 there: three-state `planned → submitted → confirmed/unknown` ledger, zero automatic retries on the paid POST, reservation-based enforcement against a pre-committed immutable ceiling, provider-evidence reconciliation — the earlier "ledger-before-POST + actual-fee accumulation" sketch is superseded and must not be implemented) is a **hard prerequisite to any `purchase-missing --execute`**. Policy, in priority order:

1. **Purchase-cheapest-complete (recommended).** Buy up the completeness frontier (per the §2(b) implemented ranking) until either the target N or the cap. At a realistic 2–5 missing core docs per near-complete case, $2,250 converts roughly 150–350 cases — the cap is unlikely to bind before the candidate supply does. Requires case.dev (Branch B of the keyless-run plan). Rationale: dollars buy N exactly where N is the binding scientific constraint; the marginal case costs ~$6–15.
2. **Reduced packet — rejected for filed core roles.** Running a case with a docketed-but-missing opposition or missing MTD memorandum changes the construct ("forecast from the record the judge saw" becomes "forecast from part of it") and creates an availability-correlated ablation nobody asked for. See §0 for the precise definition. A missing *reply*, or a genuinely unopposed motion, is acceptable.
3. **Free-only fallback (no case.dev):** take N₀, report it honestly under the cadence rules (§4), and leave the collection window open longer to grow N₀. Do not stretch the packet-completeness definition to inflate N.

**Selection-hygiene rule, restated again (rev 2's per-batch purchasing is withdrawn):** the review caught that per-batch purchase slices conflict with the frontier's own logic — a more expensive Batch 001 candidate can consume budget that a cheaper Batch 002 candidate deserved, so per-batch purchasing silently degrades "cheapest-complete" into "cheapest seen so far." Accepted. Under the §5 two-phase restructure this resolves cleanly:

- **No purchases until the candidate window closes.** All batches run acquisition-only (discover → plan → download-free → bridge-no-purchase → filter). Purchases happen **once**, globally, from the merged cross-batch frontier table, under the pre-committed rule (buy-cheapest-complete up to the cap / target N, ≤k threshold, completeness definition per §0) that John approves at work item 12 — after Batch 001's measurement, before any purchase.
- The unchanged invariant: purchase decisions are made **before any labels or model outputs exist for the affected cases** — trivially satisfied now, since labeling runs once, after the purchase phase, in Phase 2. The ledger's timestamps make the ordering auditable.
- Cost of this correctness: purchased documents arrive weeks after early batches' discovery. That is fine — purchase latency is not on the critical path until Phase 2, and case.dev purchases are minutes, not weeks.

## 4. What N is scientifically defensible (case-clustered micro-Brier)

The repo's own cadence/power machinery (`legalforecast/reporting/cadence.py`) is the authority; numbers below use its constants (paired per-case model-difference SD = 0.05, α = 0.05 two-sided, power = 0.80; n = ((1.96 + 0.8416) · 0.05 / MDE)²).

**Unit correction (review point accepted): the pipeline's unit is the docket, not the motion.** Discovery fixes both `candidate_id` and `case_id` to the docket ID (`courtlistener_acquisition.py:401-423`), motion linkage can attach multiple MTD entries to one candidate (`selection/motion_linkage.py:156-168`), and corpus readiness counts unique candidates (`corpus_readiness.py:85-106,211-239`). "150 clean motions" and "150 clean candidates" are different targets whenever a docket carries multiple qualifying MTDs. Resolution required in the §0 decision bead, recommended: **constrain each candidate to one selected target motion — which requires building a deterministic selector, not just asserting an invariant (rev 7; the fifth pass is right that "the linkage machinery already selects a target" was still false against source):** `motion_linkage.py:156-168,228-244` can produce linkages containing **multiple** motion IDs for one candidate, and `courtlistener_acquisition.py:377-423` flattens all of them — nothing selects. Work item 1 therefore includes (i) a **deterministic target-motion selector** (recommend: the earliest-filed eligible MTD whose first written disposition is on/after the anchor; ties broken by lowest docket-entry number — any rule works as long as it is deterministic, documented, and committed in `cohort-policy.json` before Batch 002), and (ii) an **enforced one-motion-per-candidate invariant** validated at corpus readiness and re-checked at freeze — a candidate reaching either gate with ≠1 selected target motion fails closed. Then candidate = docket = one motion = one cluster and the table below reads as written. If John instead wants multiple motions per docket, every "clean motions" row below must be relabeled *clean docket clusters* and the power arithmetic redone at the cluster level (motions within a docket are not independent). The epic's "150 clean motions" language should be updated to match whichever is chosen.

**Classification depends on `cycle_series`, not just N.** A run in the rapid series stays `rapid_provisional` no matter how large N gets (`cadence.py::_classification`; `CycleSeries.RAPID` vs `OFFICIAL`); official-descriptive additionally needs ≥400 prediction units. ## 5. Operational runbook — two phases, restructured (rev 3)

Rev 2 ran the full pipeline (parse → unitize → label → review → packets) inside every batch. The review's structural objection is accepted in full: per-batch downstream processing duplicates LLM billing across batches, defers cross-batch dedup until after money is spent, conflicts with a cycle-wide label audit (the sample universe doesn't exist until all batches land), and — decisively — **cannot be assembled afterward**: `acquisition merge-artifacts` merges a fixed list of downstream artifacts and omits screened cases, discovery exclusions, discovery summaries, paid-gap plans, budget plans, and purchase results entirely (`cli.py:2793-2811`), while `finalize-corpus` accepts exactly **one** screened-cases file, one discovery summary, and one exclusion ledger and enforces exact counts and unique candidate IDs (`cli.py:1515-1579,3837-3928,4111-4185`). There is no batch-aware discovery merge and no aggregated-summary schema; rev 2's "merge rehearsal" would have rehearsed software that does not exist. The restructure:

**Phase 1 — acquisition only, repeated per batch (cheap, no LLM spend, no disclosure):**

Per-batch sequence, each in its own root `tmp/acquisition/batch-NNN/` (deterministic filenames overwrite on shared roots):

1. **Discover/screen:** `acquisition discover-courtlistener --live --execute …` with order-neutral per-term caps (§2c). **Window semantics need a small CLI change first (review point accepted):** today one lower-bound date drives both the search window and contamination eligibility (`cli.py:900-949`; `courtlistener_acquisition.py:121-170,292-300,573-575`) and there is no upper bound — so a later batch can neither "sweep only the new window" (advancing the date moves the eligibility anchor) nor bound its overlap (keeping it re-searches everything). New bead: split immutable **`--eligibility-anchor`** (2026-06-30, never changes) from **`--search-window-start/--search-window-end`**, with later batches sweeping `[last_batch_snapshot - overlap, now]` — the overlap (recommend 7-14 days) catches late-indexed decisions, and the cross-batch dedup absorbs the resulting re-hits.
2. **Plan free downloads:** `acquisition plan-public-downloads --execute --screened-cases … --raw-html-dir … --target-clean-cases <nonbinding>`.
3. **Download free:** `acquisition download-free --execute --live-public-download`.
4. **Bridge + filter, no purchase:** `acquisition bridge-pacer-gaps --execute --live-case-dev` (or the Branch A N₀-only fallback per §2 step 4) → `acquisition filter-core-documents --execute` → dry-run `acquisition plan`. Output: this batch's frontier inputs. (Rev-6 regression fix, fourth pass: rev 5 dropped `--execute` from steps 2–4 here even though the preamble's command-hygiene corollary and the Batch 001 procedure already required it — dry-run forms write placeholders, `cli.py:2288-2310,2487-2560,2592-2685`.)
5. **Snapshot:** the batch root is now **immutable**. Per-batch acceptance: run cards green, exclusions reconcile against summary counts, zero paid activity.

**Cross-batch state (new engineering, the review's top-leverage item, accepted):** a **cycle candidate ledger** — which **is** the keyless plan §4b cycle acquisition store (rev 4 there merged the two into a single cycle-scoped SQLite store with batch/observation/cursor/snapshot tables; there is exactly one store, not a per-run database plus a separate ledger), keyed by docket ID, recording which batch first saw each candidate, its current state, and pointers into batch snapshots.

**Candidate-state taxonomy (rev 6, tightened in rev 7 — this is the definition the keyless plan §4b references):**

- **Immutable reason codes** (never refreshed within the cycle): construct exclusions that genuinely cannot flip with new information — `DECISION_BEFORE_RELEASE_ANCHOR` (a documented pre-anchor disposition cannot be un-filed) and non-civil/metadata screen failures. A docket in one of these states is skipped-with-record in every later batch.
- **Refreshable reason codes** (re-eligible in a later batch's window): temporary API/HTML unavailability, parse failures (`courtlistener_acquisition.py:226-319` — conditions the code already distinguishes), multi-page-docket exclusion *if* the §7 Q5 decision later flips, "screened but free documents incomplete" (free availability changes as RECAP users upload — the delayed global purchase makes this common), **and strict-clean screen failures (rev 7 — the fifth pass is right that rev 6 wrongly listed these as immutable: the strict-clean decision is derived from the presently indexed docket entries (`mtd_acquisition_screen.py:378-414`; `courtlistener_acquisition.py:336-347`), and a later-indexed entry can change the result — a strict-clean failure is re-examined when a later batch's snapshot shows new docket content, and only then).**
- **Bot challenges are not in this taxonomy at all (rev 7 — rev 6 listed "bot-challenge aborts" as a refreshable per-docket code, contradicting the keyless plan §4c, which is correct):** a confirmed bot challenge is a **run-level infrastructure stop**, never a per-docket disposition. A candidate mid-flight when the run stops simply has **no observation recorded for that batch** and keeps its prior state; nothing bot-related is ever written into a candidate's reason-code history.
- **Observations are versioned, never overwritten — with per-reason transition rules, not generic last-write-wins (rev 7; the fifth pass is right that "latest observation wins" lets a transient refresh failure silently supersede an evidenced acceptance):** each batch that re-examines a refreshable candidate appends a new observation row, and the store computes a **canonical current state** per candidate under an **evidence-precedence rule keyed by reason class**: (i) a *transient-failure* observation (fetch error, parse failure, unavailability) **never supersedes** an evidenced state (an acceptance backed by validated downloads, or a construct exclusion backed by parsed docket content) — it appends, flags the candidate for retry, and leaves the canonical state unchanged; (ii) an *evidenced* observation supersedes an equal-or-weaker prior state (acceptance → strict-clean failure on a newly parsed fuller docket is legal, and vice versa); (iii) among equal-evidence observations, latest wins. The transition table (which reason class may supersede which) is part of the store schema and gets its own tests. The batch-aware merge exports the canonical current state.
