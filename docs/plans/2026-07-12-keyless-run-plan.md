## 0. Naming and scope: this is not a "keyless" plan

The original title said "keyless." That was misleading and the reviewer was right to flag it. Precisely: **the free acquisition branch needs no paid court-data vendor credential** (no case.dev, no direct PACER). The end-to-end Cycle 1 pipeline still requires, per stage:

| --- | --- | --- |
| Discovery/screen (`discover-courtlistener --live`) | `COURTLISTENER_API_TOKEN` (free) | Enforced at `cli.py` ("COURTLISTENER_API_TOKEN is required with --live") |
| Free document download (`download-free`) | none | Allowlist covers `courtlistener.com`/`storage.courtlistener.com` — but see §4b: today only the *initial* URL is validated on the HTML path and only the *final* URL on the PDF path; redirects in between are followed unvalidated. Low impact (no credential attached), but the hardening item validates every hop before "allowlisted-only" is an honest claim. |
| Paid-gap bridge (`bridge-pacer-gaps --execute --live-case-dev`) | `CASE_DEV_API_KEY` | **Rev-6 correction (fourth pass): an executing bridge requires a fixture or `--live-case-dev` with the key** (`cli.py:1019-1024,2686-2703`) — Batch 001's Branch B needs the key here, before any purchase; without it, run the cohort doc's N₀-only fallback (§2 step 4 there) |

Each stage should run with **only its own credentials injected** — and "stage-scoped" means **per-command subpaths, not one acquisition subtree**. Rev 2 proposed a single `/agents/sandbox/legalforecastbench-acquisition` path; that would co-inject CourtListener, case.dev, and (if migrated) Firecrawl keys into every acquisition command, which contradicts the scoping rule two sentences earlier. Layout: `…/legalforecastbench/discovery` (CL token only), `…/legalforecastbench/purchase` (case.dev only), `…/legalforecastbench/firecrawl` (Branch C only), plus the parser/labeling/eval paths named in §4f. "Keyless" claims in prior drafts should be read as "free-branch needs no paid court-data key" and nothing more.

## 2. The credential branch decision

### Branch A (default, recommended): free CourtListener API token

- **What it unlocks:** the entire merged discovery/screen path plus RECAP fallback document metadata.
- **Quota: do not bank a number.** The first draft cited ~5,000 queries/day; CourtListener's own tracker ([freelawproject/courtlistener#7200](https://github.com/freelawproject/courtlistener/issues/7200)) says the published limit was 5,000 requests **per hour** and that the membership transition will change it. Any banked figure will be wrong by execution time. The engineering answer is generic: header-aware throttling, `Retry-After` honoring, and durable request accounting (§4), not a preflight built around a hard-coded quota. For sizing intuition only: a 3,000-candidate sweep needs roughly `len(query_terms) × pages + candidates × 1 docket GET` ≈ low thousands of calls.
- **John actions (blocking for live acquisition, ~10 minutes total):** (1) create/confirm a CourtListener account and API token; (2) approve the Infisical Agent Sandbox subtree so agents can consume it (`LegalForecastBench-yr43.27`, then `yr43.8` closes): per-command discovery path per §0 (e.g. `/agents/sandbox/legalforecastbench/discovery`), key `COURTLISTENER_API_TOKEN`; runtime prefix `infisical-agent-sandbox run --path <discovery path> -- …`. Stage-scoping rule (§0): this path holds **only** the CL token — case.dev, Firecrawl, parser, labeling, and eval keys each live in their own subpath.
- **Rationale:** zero engineering, zero marginal cost, supported API rather than scraped markup, and the code is merged and tested (895-test suite green per `yr43.4` close note). Strongest rejected alternative: building the Firecrawl path first "to be safe" — rejected because it spends the scarce pre-Monday frontier budget on insurance for a failure mode that has not occurred, while the actual blockers are John-action credentials and the §4 hardening.

### Branch B (if the case.dev key resurfaces): case.dev = purchase capability, not discovery

- If John still has case.dev access, do **not** revert discovery to `search_docket_entries` — the CourtListener path is now the tested Cycle 1 producer, and the 05-18 smoke showed case.dev search recall was poor anyway (4 hits, 1 unique case, and a `CaseDevResponseError` on retrieval).
- case.dev's remaining unique value is `case_dev_purchase.py`: guarded PACER purchase for missing core briefing ($3.05/doc estimate, $2,250 projected cap, ≤24 missing-core docs/case — `missing_core_budget.py:14-16`). Without it, the cohort is free-RECAP-only, which caps clean-complete yield hard (see companion cohort doc). `acquisition recover-purchased` (new in PR #36) bridges purchased docs back into parser-ready manifests; `merge-download-manifests` then merges free + purchased for parse planning.
- **Purchase-stage hardening is a prerequisite to spending money** — the binding design is §4e (rev 4): three-state ledger, reservation-based cap against a pre-committed immutable ceiling, **zero automatic retries on the non-idempotent purchase POST**, and provider reconciliation for unknown outcomes. (Superseded sketches from earlier revisions — "ledger before POST + actual-fee accumulation" — must not be implemented; today the $2,250 check is projected-only and the HTTP client silently auto-retries the paid POST, both of which §4e fixes.)
- **John action — but only after bead 7a lands (rev 7; fifth pass, accepted as high-severity security: the smoke was sequenced before the redirect hardening it exists to justify).** The smoke uses `CaseDevClient` (`case_dev_smoke.py:196-220`), whose transport today follows redirects with default `urllib` handling while carrying the bearer token (`case_dev_client.py:178-238,427-495`) — so the *first live credential test* would exercise exactly the credential-leak surface this plan says must block live Branch B. The case.dev-transport authenticated-redirect hardening is therefore a **no-dependency prerequisite (bead 7a) to any live smoke, bridge read, or purchase call** — not a consequence of the smoke saying "yes". Then: check whether the old key still authenticates. The implemented command is the **top-level** `legalforecast case-dev-smoke --live --output tmp/case-dev-smoke-recheck.md` with tight `--per-query-limit`/`--candidate-retrieval-limit` (there is no `acquisition smoke` subcommand; `CASE_DEV_LIVE_TESTS` is a test-suite switch, not a CLI contract). **Scope correction from review: the smoke proves authentication, not purchase capability.** It exercises docket search/retrieval only (`case_dev_smoke.py:196-220`); `purchase-missing` separately defaults capability to `unknown` and the purchase client refuses unless the operator asserts `document_level_purchase` (`cli.py:1085-1107`; `case_dev_purchase.py:121-165`). So Branch B has **two gates**: (i) smoke passes = the key lives; (ii) a **non-charging capability verification** = confirm with case.dev (account docs, support, or a provider endpoint that answers without buying anything) that document-level PACER purchase is enabled on this account, and record the basis for asserting the capability flag in the bead. Only then is `purchase-missing` armed. If the key is dead: decide whether to re-subscribe (only if the purchase budget will actually be spent) — otherwise run Cycle 1 free-only at reduced N.

### Branch C (fallback, build only on trigger): Firecrawl

Trigger conditions — build this only when one fires: (a) `LiveCourtListenerDocketHTMLSource` starts returning **marker-confirmed bot-challenge responses** at production volume (body/content-type challenge markers per the single §4a/§4c classification policy — a persistent *unmarked* 403 is an authorization/HTTP failure to investigate with CL, and a persistent unmarked 429 is throttle tuning, neither alone a Branch C trigger); (b) CL revokes/limits the token or the search API; (c) docket pages start requiring JS rendering that breaks `parse_courtlistener_docket_html`.

Design (banked now so Opus/Codex can build it in ~half a day):

1. **`FirecrawlCourtListenerDocketHTMLSource`** — new module `legalforecast/ingestion/firecrawl_html_source.py`, implementing the existing `CourtListenerDocketHTMLSource` Protocol. `fetch()` POSTs to the Firecrawl scrape endpoint requesting raw HTML and returns it (the parser needs `#docket-entry-table` markup; markdown output is useless here). **Pin the API version at build time**: current official docs lead with `/v2/scrape`; `/v1` appears only in migration docs. The builder must capture a live response, freeze it as a fixture, and validate the adapter against that captured schema — do not trust this doc's memory of the request shape. Config: `FIRECRAWL_API_KEY` + optional `FIRECRAWL_API_URL` via a frozen `FirecrawlConfig.from_env` mirroring `CourtListenerConfig.from_env` (`courtlistener_client.py:56-79`). **Security constraints:** the key moves to a scoped Infisical path before any automated use (never a 1Password runtime lookup — §8B boundary); `FIRECRAWL_API_URL` must pass `http_config.validate_https_base_url` against an explicit allowlist of approved Firecrawl hosts, with **every redirect hop** validated before it is followed (checking only the final URL is too late once a bearer token rode along) — accepting an arbitrary env-provided URL while attaching a bearer token is a credential-exfiltration hazard. **Both directions need validation (review point accepted):** the adapter must also validate the *target* — each `source_url` handed to Firecrawl must be an allowlisted `courtlistener.com` URL, verified before the request is composed. Otherwise a corrupted or adversarial screened-cases input turns the paid Firecrawl account into an arbitrary-URL fetch proxy. Use `urllib` like every other client in this repo (no new deps). Wire into the CLI as `--html-source {live,firecrawl}` on `discover-courtlistener` (default `live`), keeping fixtures untouched.
2. **Multi-page dockets (decision only — no design banked):** today, dockets whose entry table paginates are excluded outright (`courtlistener_docket_more_than_one_page`, `mtd_acquisition_screen.py:356`; `has_next_page` from `courtlistener_web.py:437-444`). That exclusion is a yield tax and a mild selection bias (long dockets ≠ random). Rev 2 sketched a "concatenate entry tables" wrapper; the review is right that the sketch was misleadingly shallow — the parser locates a single `#docket-entry-table` (`courtlistener_web.py:168-180,325-357`), and real pagination needs page bounds, snapshot consistency across fetches, row dedup/ordering, and multi-page provenance. Since the decision is deferred anyway, the sketch is withdrawn: what is banked is only the **measured trigger** — Batch 001's exclusion-ledger count for this reason code tells us what the exclusion costs, and the include/exclude decision bead is escalated to John with that number. Default: keep excluding.
3. **Firecrawl-backed `FreeDocumentSource` (optional):** `free_document_downloader.py` fetches allowlisted `courtlistener.com` / `storage.courtlistener.com` PDFs with urllib and already handles landing-page→PDF resolution. Only if *those* fetches get bot-blocked, add a Firecrawl variant behind the existing source Protocol. Do not build preemptively — storage-bucket fetches have a different (lower) blocking risk than HTML pages.
4. **API-loss replacement (last resort — and bigger than a search scraper):** rev 2 claimed a search-results scraper behind `CourtListenerTransport` would leave discovery untouched. Review correction, verified: that transport also serves `/dockets/{id}/`, and discovery calls `get_docket()` on every search hit (`courtlistener_client.py:87-97,385-436`; `courtlistener_acquisition.py:145-172`) — so a search-only substitute does not survive token revocation; the docket-metadata leg dies with the same token. If the trigger is *bot-blocking of docket HTML only*, the already-designed docket-HTML adapter (item 1) is the complete answer. If the trigger is *API loss* (revocation, endpoint removal), the substitute must cover **both** search results and docket metadata — scraped search pages normalized to `search_recap_documents`' shape *plus* docket metadata recovered from the docket HTML page (which the pipeline already fetches and parses) or scraped from it. Design that full substitute only if this distinct trigger actually occurs; it is the largest and least likely branch. Probability: low — CL is friendly to programmatic access and the token is free.

Rejected alternative for Branch C: a standalone Firecrawl pipeline writing the legacy discover JSONL (the original survey design). Rejected because PR #36 moved the production entry point; feeding the legacy `discover` command would strand the output one stage before `plan-public-downloads` expects it (screened-cases schema, not raw entry JSONL) and duplicate the screening logic that `discover_courtlistener_mtd_candidates` already implements fail-closed.

## 4. Hardening the merged path — required before the first live sweep

The reviewer's premortem found the merged path is correct-on-happy-path but not crash-safe or polite at 3,000-candidate scale. The first draft treated politeness as "cheap, do in week 1" and omitted crash safety entirely; both are now **prerequisites to Batch 001**, not follow-ups. Grouped for beading:

**4a. Politeness and rate handling (blocks first live sweep):**

- **Layering correction from review:** rev 2 put the inter-request delay "in `_screen_candidate`" — wrong layer. Discovery issues search API requests, docket API requests, and separate HTML requests (`courtlistener_acquisition.py:145-172,226-267`), and the REST client's retries add more. Rate limiting and request accounting belong in the **transport/client wrappers** (`CourtListenerClient` + the HTML source), so every wire request — search pages, docket GETs, HTML fetches, retries, and failed attempts — is throttled and counted once, at the place it happens. `_screen_candidate` stays throttling-ignorant.
- Configurable inter-request delay (default ~1s) at that transport layer — 3,000 candidates at full speed is exactly how you earn the Cloudflare block that forces Branch C.
- Retry/backoff overhaul: docket-HTML fetches currently have no retry and non-404 HTTP errors abort the run; the REST client retries 429/5xx only twice with default zero backoff and ignores `Retry-After` (`courtlistener_client.py:385-403,474-506`). Add exponential backoff with jitter, honor `Retry-After`, and cover both API and HTML requests.
- **One 429/403 policy, stated once (rev 2 contradicted itself between 4a and 4c; rev 7 removes the residual contradiction the fifth pass caught — rev 6 still let an exhausted bare 403 "become" a bot challenge in 4c while saying here that a generic 403 proves nothing):** a 429 is **throttling first** — honor `Retry-After` (or backoff) within a bounded retry budget. **Bot-challenge classification requires challenge markers (body/content-type), full stop.** Budget exhaustion changes the *outcome* (the run stops) but never the *classification*: exhausted marked responses ⇒ bot-challenge stop (Branch C trigger); exhausted unmarked 403 ⇒ authorization/HTTP failure stop (investigate credentials/CL policy — not a Cloudflare verdict, not a Branch C trigger); exhausted unmarked 429 ⇒ rate-limit exhaustion stop (tune throttling). All three stop cleanly with durable state; they differ in the operator message and the trigger they feed.
- Durable request accounting: count requests per run in the run card so the operator can see consumption against whatever quota CL currently publishes. (This replaces the first draft's "fail at ~80% of the known daily quota" preflight — there is no stable known quota to preflight against, and the client does not expose enough response-header state to infer one. Generic throttling + accounting is the robust version.)

**4b. Crash safety and run state (blocks first live sweep) — transactional, not append-only:**

Rev 2 said "append screened/excluded records incrementally (JSONL append is the natural fit)." The review demolished that, correctly: every discovery invocation starts with empty results, empty `seen_docket_ids`, and cursors at `None` (`courtlistener_acquisition.py:136-150`), so a restart replays page 1 and *duplicates* whatever the appended file already held; and the existing append helper (`cli.py:6211-6218`) has no locking, fsync, checksum, or torn-tail recovery. Appending survivable bytes is not a resumability design. The design that is:

- **One cycle-scoped transactional acquisition store — not a per-run database plus a separate cycle ledger (third pass, accepted as a simplification).** A single SQLite database per cycle (stdlib, real transactions) with **batch, observation, cursor, and snapshot tables**: stable candidate key (docket ID), per-candidate *versioned observations* (which batch observed what, when), per-candidate state (`pending → screened | excluded`, with the terminal-vs-refreshable taxonomy defined in the cohort doc §5), **per-term pagination cursors**, request accounting, and run metadata. This same store *is* the cohort doc's "cycle candidate ledger" — rev 3 described them as two artifacts, which would have created synchronization work without adding any guarantee; there is exactly one store. Single-writer locking (one discovery process per store), atomic commit per candidate transition, corrupt-tail/partial-write recovery on open. Screened-cases/exclusions/summary JSONL artifacts are **exported from it** — keeping the repo's JSONL artifact contract — under the completion rules below. If the executor prefers pure JSONL they must implement locking, fsync, checksums, and torn-tail recovery themselves — i.e., re-implement the cheap half of SQLite badly.
- **Completion semantics: a checkpoint export is not a completed run (third pass, accepted as high severity).** `plan-public-downloads` reads whatever screened-cases file it is given and verifies nothing (`cli.py:2487-2524`), so a crash after a checkpoint can leave a partial canonical JSONL that downstream stages mistake for a finished discovery. Required: (i) checkpoint exports are published under **partial/snapshot-specific names** that no downstream command accepts; (ii) canonical artifact names are **atomically published only from a `complete = true` completion snapshot** recorded in the store (all terms exhausted or caps bound, all candidates in terminal or explicitly-deferred states); (iii) the completion snapshot's hash goes into the run card, and downstream stages (`plan-public-downloads` first) gain a **required verification** that their input matches a completed run's snapshot/config hash — a downstream completeness gate, not a convention.
- **Run-configuration identity binds resume — at two levels (rev 6, fourth pass accepted: a single digest containing search-window bounds cannot stay constant while later batches intentionally change those bounds).** Resuming into a store whose configuration differs from the run that wrote it silently mixes incompatible sweeps. The store records: (i) an **immutable cycle-policy identity** — eligibility anchor, query terms and their order, screening-configuration hashes, store schema version — fixed for the life of the cycle store; any run whose cycle identity mismatches is refused outright; and (ii) a **per-batch config digest** — search-window bounds, per-term scheduling/caps, page size, candidate/target limits — recorded per batch; resume *within a batch* refuses a per-batch digest mismatch (new batch or explicit migration instead), while a new batch legitimately carries a new per-batch digest under the same cycle identity.
- **Cursor transaction rule, stated formally:** a term's pagination cursor advances **only after every hit on that page is durably represented** in the store (as an observation, at minimum `pending`). An early cursor commit skips candidates on crash; a late commit is safe only because replay deduplicates against recorded observations — the store's dedup-on-replay behavior is therefore part of the contract and gets its own test.
- Atomic writes everywhere in discovery: raw HTML is currently written directly to its final filename and any existing file is trusted on resume with no hash, size check, or parse-success marker (`courtlistener_acquisition.py:255-267`). Write to temp file → validate (non-empty, parseable) → hash → atomic rename; record hash + retrieval timestamp in the state store.
- Cache freshness policy: a reused docket snapshot may predate newly available documents. Record retrieval timestamps; per-batch snapshots are immutable; refresh eligibility follows the **immutable-vs-refreshable reason-code taxonomy the cohort doc §5 now defines** (rev 6: that definition previously did not exist where this doc pointed) — never within a batch; across batches only for refreshable states, via versioned observations and the current-state export.
- Free downloads: `free_document_downloader.py` reads responses fully into memory and writes directly to final paths; resume accepts any existing file without re-validation (`free_document_downloader.py:125-157,263-297`); the manifest lands only after all downloads. Switch to bounded streaming → temp file → content validation → hash → atomic rename → per-item checkpoint (same state-store pattern).
- Redirect validation on every hop (credential-free paths): docket HTML validates only the initial URL and then lets `urllib` follow redirects; the PDF path validates the final URL after opening it (`courtlistener_acquisition.py:77-92`; `free_document_downloader.py:118-152`). No credential is attached so impact is bounded, but "allowlisted-only" must mean each hop is checked against the allowlist before it is followed.
- **Authenticated redirect handling — the higher-risk clients were outside the hardening scope until rev 6 (fourth pass, accepted as high-severity security):** `UrlLibCourtListenerTransport` validates only the initial base URL and then uses default `urllib` redirect handling while the client attaches `Authorization: Token …` (`courtlistener_client.py:244-289,474-512`); `UrlLibCaseDevTransport` does the same with `Authorization: Bearer …`, **including on the charge-bearing purchase POST** (`case_dev_client.py:178-238,427-495`). Default `urllib` redirect handling copies non-content headers — including `Authorization` — into the redirected request, so a cross-host 301/302 leaks the token to an arbitrary host; a 301/302/303 can also silently convert the paid POST into a GET. Required: (i) on every authenticated request, each redirect hop is validated as **same-host HTTPS** before it is followed; (ii) any host change strips the credential (and, for this repo's threat model, simply fails the request — there is no legitimate cross-host redirect for these APIs); (iii) the **paid purchase POST rejects redirects entirely** (any 3xx is a terminal `unknown` outcome for the §4e ledger, full reservation retained), unless case.dev documents server-enforced safe redirect semantics for it. CL-client scope rides with bead 3 (blocks the first live sweep — the token flows on every discovery request); case.dev-client scope is **bead 7a — a standalone no-dependency prerequisite that blocks any live case.dev call at all, the §2 Branch B auth smoke included** (rev 7: rev 6 had it inside bead 7, which depended on the smoke saying yes — backwards). **Scope correction (rev 7; fifth pass, accepted — a rev-6 claim was false):** `recover-purchased`'s own download path **already** validates redirect targets and strips authorization on cross-host redirects (`purchased_document_recovery.py:61-82,112-175`) — do not re-implement that. Its actual remaining defect is **crash safety**: it writes documents directly to their final paths and publishes the recovery manifest only after the whole run (`purchased_document_recovery.py:380-447`; `cli.py:3271-3284`), so a crash mid-recovery leaves unvalidated partial files at canonical paths and no manifest. Apply the same temp-file → validate/hash → fsync → atomic rename → per-item checkpoint protocol as the free downloader to purchased recovery (bead 7 scope).
- Response-size and disk ceilings: docket HTML and PDFs use unbounded `response.read()`. Add byte ceilings, enforce Content-Length when present, and a pre-run free-space check.

**4c. Bot-block signal (blocks first live sweep, feeds the Branch C trigger):**

- `LiveCourtListenerDocketHTMLSource.fetch` currently maps any non-404 HTTPError to a generic `CourtListenerClientError`. Add explicit handling per the single 429/403 policy in 4a: a response is classified `CourtListenerBotChallengeError` **only when the body/content-type matches challenge markers** — never from status code + persistence alone (rev 7 fix: rev 6's "or when 403/429 persists after the bounded retry budget" contradicted 4a's own rule; an exhausted unmarked 403 stops the run as an authorization/HTTP failure, an exhausted unmarked 429 as rate-limit exhaustion — distinct operator messages, neither a Branch C trigger). **REST-transport scope (rev 7; fifth pass, accepted):** the classification must run over the **bounded raw body before JSON decoding** — today `UrlLibCourtListenerTransport` decodes JSON first, so an HTML challenge response dies inside JSON parsing before any classifier could see it (`courtlistener_client.py:284-289,566-570`); the transport reads a bounded body, classifies challenge markers, and only then attempts JSON. Rides with bead 3 (transport) + bead 6 (stop semantics). **Semantics (carried from rev 2):** a confirmed bot challenge is a run-level condition, not a property of one docket — do *not* route it to the per-docket exclusion ledger (and note `_screen_candidate` catches `CourtListenerUnavailableError`, so an uncaught new error type would abort anyway, just illegibly). **4e. Purchase ledger and spend cap (blocks any purchase — Branch B gate, not Batch 001):**

Rev 2 proposed "append a ledger record before each POST; on resume, skip anything already ledgered." The review is right that this is actively unsafe in both directions: crash *before* the POST and the resume logic permanently suppresses a purchase that never happened; crash *after* the charge but before the receipt persists and the outcome is ambiguous with money already spent. Today purchase attempts exist only in memory until the result returns (`case_dev_purchase.py:193-240`). Required design:

- **Zero automatic retries on the paid POST — this comes first (third review pass, accepted as critical).** The state machine below controls what happens *around* a call to `purchase_pacer_document()`, but the call itself currently rides the generic `_request_json()` path, which **auto-retries every 429 and 5xx up to two times and represents timeouts as retryable synthetic 504s** (`case_dev_client.py:427-475,528-532`), and the request carries no idempotency key — only `live` and `acknowledgePacerFees`. One `submitted` ledger row can therefore trigger up to three charge-bearing POSTs before the ledger ever sees control again, and a successful charge whose response is lost gets immediately re-POSTed. Required: the purchase call uses a dedicated request path with **retries = 0**, unless case.dev's *current* API contract confirms a server-enforced idempotency key — in which case the ledger's operation key is transmitted as that idempotency key and reused across every attempt and restart. Timeout, connection loss, malformed response, or post-send 5xx classify the row as `unknown` (full reservation retained), never as retryable.
- **Three-state transactional ledger:** `planned → submitted → confirmed | failed | unknown-needs-reconciliation`. `planned` records intent (docket, document, projected fee); the transition to `submitted` — carrying a client-generated operation key — commits *immediately before* the POST; `confirmed` commits with the response (actual `pacer_fees`, receipt fields). On resume: `planned` rows are safe to submit; `confirmed` rows are done; `submitted`/`unknown` rows are **reconciled against authoritative provider evidence before anything else runs**. Reconciliation caveat (third pass, accepted): the current client exposes `get_document()` and the purchase POST but **no receipt, operation, billing, or purchase-history lookup** (`case_dev_client.py:423-439`) — "the document is now available" does not prove who paid or how much was charged. The executor must identify a real reconciliation source (case.dev billing/receipt endpoint, account statement export, or support confirmation) and record it; **a human may not release an `unknown` reservation merely by declaring the row closed** — closure requires recorded provider-side evidence (or an explicit written-off reservation that stays counted against the cap for the rest of the cycle). "Ledger row exists" never means "purchase succeeded"; the state does.
- **Cap enforcement via reservation, not post-hoc summation — and the reserve amount must be a verified worst case, not the configurable default.** Actual `pacer_fees` are learned only from the successful response (`case_dev_purchase.py:262-310`), so summing actuals cannot stop the request that crosses the cap. Enforce with a conservative **pre-submission reservation** released down to actuals on confirmation. Third-pass corrections, all accepted: (i) `$3.05` is `DEFAULT_PURCHASE_COST_USD`, a configurable estimate (`missing_core_budget.py:14-16`), and both the per-document cost and total budget are operator-overridable CLI arguments (`cli.py:872-879`) — a default is not a ceiling. (ii) Responses carry separate PACER and service-fee components (`case_dev_purchase.py:300-310`), so the reserve must cover the *sum including service fees and rounding*; before the first live purchase the executor verifies the true per-document worst case against case.dev's current fee schedule (or a quote endpoint if one exists) and pins that number in the cycle policy artifact. (iii) **Missing or malformed fee fields currently normalize to `"0.00"`** (`case_dev_purchase.py:300-318`) — that must fail closed: a response without parseable fees keeps the full reservation and marks the row `unknown`, never confirms a zero-cost purchase.
- **One ledger, cycle-global — with an enforced canonical identity, not a convention.** The cap is a cycle property; per-batch ledgers (or two operators with separate output roots) can each independently stay "under cap" while jointly exceeding it, and today `purchase-missing` is output-root-scoped (`cli.py:3142-3200`) so nothing prevents that. Required: `purchase-missing` gains a **required ledger argument** bound to a pre-committed **cycle policy artifact** (immutable cycle ID, canonical ledger locator, and one immutable hard cap that caller-supplied budget-plan values cannot raise). The ledger is exclusively created and locked (single writer); the command rejects any conflicting ledger path or cycle identity. Belt-and-suspenders: a provider-side spending limit on the case.dev account if the dashboard offers one. (The cohort doc's global-purchase phase means there is exactly one purchasing session in practice — this makes that a machine-checked invariant instead of a hope.)

**4e′. Provider-call journaling for LLM stages (blocks unitize/label on real cases — free-only branch included):**

Rev 2 bundled model-POST idempotency into the case.dev bead, making it conditional on Branch B. Wrong conditionality (review point accepted): `llm-unitize`/`llm-label` accumulate results in memory, re-raise on a late exception, and write outputs only after the whole function returns (`llm_pipeline.py:180-307`; `cli.py:3494-3508,3566-3583`) — so a crash at document 199 of 200 rebills 199 successful calls on rerun, and that is true with zero case.dev involvement. Separate bead, unconditional. **Journal unit = the individual provider HTTP attempt, and completeness = result-reconstructible (rev 7; the fifth pass showed rev 6's grain was still wrong in both directions):** rev 6 journaled one logical `(stage, candidate, judge)` row with status/cost — but (i) that is not the billable operation: a single logical call makes **up to three HTTP attempts** (`complete_live_prompt()`, `live_model_solver.py:41-43,153-206,646-676`), each of which can bill for inference that produced no usable result; and (ii) a "complete" status row is **not resumable**: if judge 1 succeeds and judge 2 fails, judge 1's output, labels, and votes exist only in local collections until the whole ensemble completes (`llm_pipeline.py:323-367,403-456`) — a rerun that "skips complete rows" has no vote from which to rebuild the ensemble. The durable row therefore records, per attempt: **input + prompt digest; registry/model-policy hash; attempt identity (logical-call key + attempt ordinal); a pre-call reservation against the cycle ledger; the raw and normalized response; the reconstructed labels/votes for that judge; actual token/cost settlement; committed transactionally** (result + reservation settlement in one commit — same 4b state-store pattern). Replay policy: a rerun reconstructs completed judges' votes *from the journal* and resumes the ensemble mid-flight, calling only the judges (and attempts) that never settled. **Reservations draw on the one cycle-wide per-provider/account ledger shared with the eval stage** (eval-readiness §4a rev 5 — two independently-capped stage ledgers can jointly exceed the approved cycle ceiling; either one ledger or frozen sublimits whose sum ≤ the cap). Per-stage cost accounting reports against that ledger. The eval solver's own attempt policy is parameterized in the eval-readiness doc (§4a there); the journal row contract here is shared by both.

**4f. Parser subprocess environment (blocks parse stage):**

- `mistral_markdown_parser.py:114-145` launches the external parser checkout with `os.environ.copy()`. Under a single broad Infisical environment that would hand CourtListener, case.dev, provider, and any other ambient secrets to a subprocess that needs exactly one credential. Run the parse stage under its own Infisical path — but a stage-scoped path alone is **not** the fix: it neither removes unrelated inherited secrets nor closes the fallback below.
- **The 1Password fallback must be made impossible, not just unlikely (rev 7; fifth pass, accepted as high-severity security):** the external parser explicitly calls `op read` when `MISTRAL_API_KEY` is absent (`~/Development/tools/parser/src/parser/pdf.py:679-688`; `…/utils/helpers.py:19-57`) — and since `PATH` stays in any workable child env, a missing or misinjected Infisical variable **silently crosses the "no 1Password at runtime" §8B boundary** instead of failing. All three required: (i) the child environment is **built from an explicit allowlist** (parser key + `PATH` + locale + the minimum the parser demonstrably needs), never by copying-and-pruning ambient `os.environ`; (ii) the parent **validates `MISTRAL_API_KEY` is present and nonempty before spawn** and fails the parse stage closed otherwise; (iii) the parser checkout is **pinned to a reviewed revision whose behavior on a missing key is fail-closed** — no `op` attempt — either because that revision supports an explicit no-fallback mode or because the pin postdates a parser-side fix (coordinate the small upstream change; the pin is what makes it load-bearing). Acceptance includes a **negative test that places a sentinel `op` executable on `PATH`** and proves the parse stage (a) succeeds with the key present without invoking it and (b) fails closed with the key absent without invoking it.
