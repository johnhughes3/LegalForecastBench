## 0. HARD BLOCKER: the required matrix cannot be dispatched as one run — and rev 2's shards would be rejected too

The 256-cell finding stands: the official workflow hard-fails above **256 matrix cells** (`.github/workflows/run-benchmark.yaml:627` sets `MATRIX_LIMIT: "256"`; the matrix step raises at `:845-849`), and Cycle 1's 150 cases × 4 models × 2 ablations = **1,200 cells** is 4.7× over. But the second review pass showed rev 2's fix — eight one-model/one-ablation dispatches against the same freeze — **fails before reaching the matrix, twice over**:

1. **The provenance gate rejects partial model sets.** `build_dispatch_provenance()` requires the requested model keys to equal **exactly** the set of models introduced by the current freeze; a one-model request against a four-model freeze raises before matrix construction (`legalforecast/publication/dispatch_provenance.py:108-115`, invoked at `run-benchmark.yaml:586-600`, covered by `tests/test_dispatch_provenance.py:96+`). Fan-out is not free: it needs a **shard-aware provenance protocol** — the freeze declares the full model set *and* an explicit shard schedule, and the gate accepts a dispatch that matches a declared shard.
2. **Every successful shard would then auto-trigger an invalid partial aggregate.** `aggregate-results` runs automatically after each successful matrix (`run-benchmark.yaml:1022-1029`), passes the full four-model registry (`:1233-1270`), and `official_aggregate` derives all registry model keys and **rejects missing cells** (`official_aggregate.py:1185-1243,1407-1462`) — and each partial run can sync into the same report prefix (`run-benchmark.yaml:1284-1291`), so even the failure mode pollutes canonical outputs.

**Corrected sharding design (bank this):**

- Shard the official run into **one dispatch per (model_key, ablation) pair**: 150 cells each, under the limit; 8 dispatches against the same frozen cycle artifacts. Unchanged from rev 2.
- **New: an explicit `shard-only` execution mode.** A shard dispatch runs its matrix, writes its disjoint cells into the per-case S3 union (`s3://…/per-case/${CYCLE_ID}` — cell-level resume there is reviewer-verified solid), and **stops**: no aggregation, no report sync, no publication. Concretely: a workflow input (recorded in provenance) that gates the `aggregate-results` job off.
- **New in rev 5: an explicit concurrency policy — without it most shard dispatches get cancelled before they run (fifth pass, accepted as a blocker).** The workflow's concurrency group is `run-benchmark-${cycle_id}-${ref}` with `cancel-in-progress: false` (`run-benchmark.yaml:138-140`). All eight shard dispatches share one cycle ID and ref, and GitHub Actions keeps **one running and, by default, one pending run per group — a newly queued run replaces the prior pending run** (it does not queue behind it; `cancel-in-progress: false` does not preserve multiple pending runs). Dispatching all eight shards therefore silently cancels up to six of them. The execution policy (§4, bead 1a) must pin one of: (i) **shard identity in the concurrency key** — e.g. `run-benchmark-${cycle_id}-${ref}-${model_key}-${ablation}` — with genuinely concurrent semantics for everything shards share (the bead-1e reservation ledger and circuit breaker are already designed for concurrent callers, so this is consistent); (ii) `queue: max` on the existing group, deliberately serializing shard runs (slower, simpler contention story); or (iii) one orchestrator workflow that owns the eight-shard sequence (most machinery). **Recommend (i)**: shards are cell-disjoint by construction, the S3 union is cell-keyed, and the spend controls are the only genuinely shared mutable state — and they are being built concurrency-safe anyway. Whichever is chosen: the workflow computes its concurrency key from the dispatched shard identity, and a CI assertion verifies the key expression includes the shard fields (a regression here re-introduces silent cancellation). This lands in bead 1b, because it controls attempt ordering, ledger contention, receipt production, and fan-in completion.
- **New: shard-aware provenance.** The freeze binds a **compact shard schedule** — the allowed (model_key, ablation) shard list, repeat policy, and dispatch/attempt identity rules — and the provenance gate validates each dispatch against a declared shard instead of demanding the full model set. This extends the freeze schema (see §4 — the freeze enum has no slot for it today) and `build_dispatch_provenance`.
- **Immutable shard-completion receipts — the authoritative fan-in protocol (rev 4; rev 5 makes it executable — the fifth pass is right that "the shard job writes exactly one receipt" named a job that does not exist, and that the rerun rule was not a selection algorithm).** Current provenance records workflow run/attempt, freeze hash, and model keys, but **not** ablation, successful completion, labels/manifest hashes, attempt-policy hash, or any commitment to the result objects (`dispatch_provenance.py:117-168`); each workflow uploads its provenance as a per-run artifact (`run-benchmark.yaml:865-873`) and today's aggregation downloads only the *current run's* artifact (`:1177-1181`) — nothing ties union objects to an accepted attempt. Three required pieces:
  1. **A real `finalize-shard` job (rev 5, bead 1c).** The workflow today proceeds from the 150 independent `run-case` matrix jobs directly to `aggregate-results` (`run-benchmark.yaml:875-1029`) — no job can observe "all cells complete" and write one receipt. Add an explicit `finalize-shard` job that runs after the full matrix (`needs: run-case` with explicit success verification, not `if: always()`-and-hope), verifies the shard's expected cell set against the frozen schedule, **distinguishes fresh cells written by this attempt from valid resumed cells adopted from a prior attempt** (both are recorded, per-cell, in the receipt), commits the **exact S3 object versions and content hashes** used by this attempt, and writes the receipt only after every check passes.
  2. **Immutable per-attempt receipts.** Receipt keys include the workflow run/attempt identity — a rerun writes a *new* receipt, never overwrites (overwrite of any receipt key is an error). Contents: the exact `(model_key, ablation)`; the execution-policy and freeze hashes; registry, labels, and frozen-manifest hashes; workflow run/attempt and repeat/attempt-policy identities; the per-cell fresh-vs-resumed record; and a **result-object hash + S3-object-version list (or Merkle commitment) over the cells it committed**.
  3. **An explicit rerun-selection algorithm (rev 5 — "reject stale, duplicate, superseded" was a wish, not an algorithm).** Fan-in selects as follows: if exactly one receipt exists per schedule-declared shard, it is accepted; if any shard has multiple receipts (a rerun happened), fan-in **refuses to proceed** until the operator commits a hash-bound **accepted-attempt map** — a small freeze-amendment artifact listing exactly one accepted receipt (by run/attempt identity) per multi-receipt shard, itself hash-linked to the freeze and recorded in the final report. Fan-in then verifies every accepted receipt's hashes against the freeze byte-for-byte, verifies the union objects (content hash *and* object version) against the accepted receipts' commitments, and rejects any union object not committed by an accepted receipt. This is execution provenance, not the rejected second completeness oracle — `official_aggregate`'s Cartesian gate still answers "is every cell present"; receipts answer "is every present cell the one an accepted attempt wrote".
- **Completeness itself still needs no new oracle (carried from rev 3):** `official_aggregate` already constructs the full case × model × ablation Cartesian product and rejects missing *and* extra cells — aggregation *is* the completeness check. Rev 2's separate 1,200-row planned-cell manifest stays cut; the compact shard schedule + receipts + the existing aggregate gate cover it.
- Per-dispatch cost caps come along for free, but see §4a — they are per-dispatch *projections*, not provider-level spend controls; six of the eight dispatches share the OpenAI account.
- The hypothetical fifth-model amendment path stays **deferred**: it cannot matter until sharded fan-in works, and the current four models do not fit without it.

This is the bead-1 family (1a–1f, split in rev 4; rescoped in rev 5) in the §5 work breakdown and it precedes freeze. The family's joint acceptance test is the smoke protocol in §4 (a shard-only dispatch on a throwaway smoke cycle, then a `--verify-only` fan-in over a complete tiny matrix — see §4; a full fan-in over a nonofficial cycle is structurally impossible by design, so the rehearsal mode is explicit, not implied).

**(b) Measured label error rate — the promised audit is not what the code does; implement the schema before labeling, sample after the cycle merges.** The first draft treated this as "operational only." Wrong. Current code (reviewer-verified): fixed default sample of 30 (`llm_pipeline.py::DEFAULT_LABEL_AUDIT_SAMPLE_SIZE`), sampled **per case-level ensemble** rather than cycle-wide, eligible units restricted to `AUTO_LABEL` decisions (`ensemble.py:336-376`), no stratum mapping (everything lands in one `"all"` stratum), and a case with no unanimous auto-labels can return `skipped` with a **null error rate**. Depending on invocation, the current defaults could also generate up to 30 human reviews *per case*.

Two rev-2 design errors, both corrected per the second review pass:

- **The rev-2 strata were not implementable against the persisted data.** (i) "Adjudication-routed" as an *error-rate* stratum is incoherent: audit comparison scores `result.auto_labels`, and a disagreement-routed unit has no selected auto-label to score (`ensemble.py:393+`). Auto-label error and adjudication workload are **different estimands** — report them separately: the error-rate audit samples auto-labeled units; adjudication-routed units get a *volume/routing* report (how many, why) plus their own human-resolution record, not a pretend error rate. (ii) A "partial" stratum cannot be recovered from persisted outcomes: `PARTIAL_DISMISSAL_ONLY` and material-survival both persist as `fully_dismissed=False` (`label_outcomes.py:267,478+`). The label schema must **retain the raw resolution** (or an explicit `audit_stratum` field) at write time — one more reason this bead precedes labeling.
- **The rev-2 gate ("any empty or undersized stratum blocks release") confused missing evidence with absent population.** A valid cycle may legitimately contain zero partial outcomes or zero judge disagreements; blocking release on that makes release impossible for reasons that have nothing to do with audit quality. Corrected semantics, predeclared before labeling: an overall sample floor (≥20 units or 10% of cycle units, whichever is larger), an allocation rule across the strata **observed in the cycle** (proportional with per-stratum minima), finite-population handling (a stratum smaller than its minimum is audited exhaustively), and fail-closed triggers = any *observed* stratum that the sampler failed to sample, any `skipped`/null-rate outcome, or a breached error threshold. An empty stratum in the population is recorded as empty, not treated as failure.

**Sequencing correction (review point accepted):** rev 2 claimed the audit "cannot be retrofitted after labels exist without replaying paid calls." Overstated — the ensemble/model outputs are retained in the audit records (`llm_pipeline.py:425+`), so *sampling* can run after all labels exist with no paid replay. What genuinely must precede labeling is the **schema** (raw-resolution/stratum field retention) and the **predeclared sampling plan**.

**Rev-4 correction: "one invocation over the merged corpus" is NOT cycle-wide sampling — the code must change (fourth pass, accepted as a labeling blocker).** Rev 3.1 implied that because labeling runs once over the merged cycle, the sample is cycle-wide "by construction." The architecture under the CLI is per-case: `llm_label_cases()` loops over selections creating **one ensemble per case** (`llm_pipeline.py:307-377`), **immediately generates audit packets for that case** (`:1007-1045`), resume re-samples each audit record's own ensemble separately (`:1195-1265`), and `sample_unanimous_labels_for_audit()` accepts exactly one `EnsembleRunResult` (`labeling/ensemble.py:336-376`). One CLI invocation is therefore many independent per-case samples with per-case floors — not a stratified draw over the cycle. Work item 3 must (i) **remove audit sampling from the per-case labeling loop entirely** and (ii) add a **separate post-label cycle-sampler command** that loads all persisted ensembles for the cycle and draws one deterministic stratified sample. Executable sampling policy, predeclared in the execution-policy artifact (§4):

- **Population:** auto-labeled units only (adjudication-routed units are the separate routing report, per above). Sample size `n = min(N_auto, max(20, ceil(0.10 × N_auto)))`.
- **Strata:** mutually exclusive over auto-labeled units — {unanimous-grant, unanimous-deny, partial} — where "partial" comes from the retained raw resolution (below), never re-derived from `fully_dismissed`.
- **Determinism — seeded from a pre-audit-immutable artifact, not from the final labels (rev 5; fifth pass, accepted as a blocker: the rev-4 seed was circular).** Rev 4 derived the seed from `labels_sha256`, but `apply_adjudicated_reviews()` **rewrites the sampled labels with their adjudicated versions** (`llm_pipeline.py:1151-1166`) — the final labels hash depends on which sample was drawn from that hash. Corrected: seed = SHA-256 of (cycle_id ‖ **ensemble-corpus hash** ‖ sampling-policy-artifact hash), truncated — where the ensemble-corpus hash is computed over the persisted per-case ensemble outputs **as they stand at sampling time, before any audit-driven adjudication rewrites them**, and is itself published (hash-committed) before the first human review begins. The final adjudicated-labels hash (`labels_sha256`) is recorded **separately** in the freeze as the post-audit output, never as a seed input. Derivable by any verifier, chooseable by no operator, and provably fixed before the audit can influence anything.

Changing the policy after labels exist requires a new artifact hash — visible, not silent.

## 3. GPT-5.6 anchor: decision record (decided and merged — recorded here so nobody re-litigates it blind)

Only dated model IDs or a provider-published immutable revision would close that; neither exists for these tiers. So: echo validation is a tripwire for announced/visible changes, the residual silent-roll risk is disclosed in METHODS, and a mid-cycle *detected* change fails closed (then decide re-anchor vs. drop).

### 4a. Cost model — corrected (first draft was wrong about what the gate prices)

The workflow's preflight prices every cell as `input_tokens × input_price + max_output_tokens × output_price` (`run-benchmark.yaml:745-761`), and **`max_output_tokens` is 128,000 for all four registry entries**. At ~120K input tokens/case, the *projected* cost the gate enforces is ≈ $2.5–3K for the full matrix (e.g., Sol's projected output charge alone is 128K × $30/M ≈ $3.84/cell — more than its input charge), even though *expected actual* spend at realistic forecast-length outputs (~2–4K tokens) is the first draft's $250–500. Both numbers are real; the first draft's error was budgeting against the wrong one — a $500 cap would fail the preflight before a single call.

Fix, in order of preference:

1. **Set `max_output_tokens` in the registry to a realistic solver bound (recommend 16,000) before freeze.** A forecast + rationale does not need 128K output. The field **does** drive the actual API request's max-output parameter for all four providers — OpenAI, Anthropic, Bedrock, and Gemini (`live_model_solver.py:397-453,484-496`; rev 5 cuts the rev-4 "verify whether" pre-task per the fifth pass — the source already answers it) — so lowering it bounds real spend too, which is strictly good. This makes projection ≈ realistic × small margin.
2. Then set `max_projected_model_cost_usd` **per dispatch** at ~2× that dispatch's projection. **These are not per-provider caps** (second pass, carried): six of the eight dispatches bill one OpenAI account; the workflow checks each dispatch's static projection independently (`run-benchmark.yaml:745-761,845+`) and nothing accumulates across dispatches — eight individually-passing dispatches can jointly spend 8× any single ceiling. **Rev-4 correction (fourth pass, accepted as critical): rev 3 called provider dashboard limits "the real cap" — for OpenAI that is false.** OpenAI's project *budget* is explicitly a **soft threshold**: it alerts, and requests continue after it is exceeded (per OpenAI's own project-budget documentation) — it is monitoring, not a spending boundary. Consequences, in order: (i) **the OpenAI path requires a mandatory atomic pre-call reservation ledger or a capped proxy in front of the API** — a shared, atomically updated ledger (or gateway) that every shard consults with a reservation *immediately before each provider call*, refusing the call when reservations would exceed the pre-committed ceiling; **timed-out or otherwise ambiguous attempts retain their reservations** until reconciled against usage records (same fail-closed shape as the keyless §4e purchase ledger). This is no longer "optional engineering" — for OpenAI it is the only rejecting control. (ii) Anthropic-side: verify at provisioning whether the workspace spend limit actually *rejects* requests (Anthropic's docs describe it as enforcing); if it does, it is a real backstop for the two Anthropic shards; record the verification in `yr43.22`. (iii) Per-dispatch projection caps remain the early-warning layer only. Unless John verifies a separately-purchasable rejecting product-level control for OpenAI, bead 1e builds the reservation ledger. **(iv) Rev 5 — one ledger per (provider, account) across the whole cycle, labeling included (fifth pass, accepted: the $1,000 ceiling in the envelope below "includes labeling headroom," but labeling and evaluation were designed against separate ledgers and stage caps — two independently-under-cap ledgers can jointly exceed the approved ceiling).** The reservation ledger is **cycle-wide per provider/account**: labeling calls (keyless §4e′ journal) and eval-shard calls reserve against the *same* ledger, or — if operationally separated — the frozen policy pins **hard sublimits whose sum ≤ the approved cycle cap**, and neither stage can raise its own sublimit. **(v) Honesty note, stated in METHODS (fifth pass, accepted):** a ledger consulted by code that also holds the raw provider key is an *accidental-overspend* control, not a privilege-enforced cap — the code being controlled could skip the check. That is the right tool for this threat model (our own bugs, not our own malice). If a literally *rejecting* boundary is ever required, a capped gateway/proxy must hold the raw key and the workflow gets only the proxy endpoint; record that as a known upgrade, not a Cycle 1 requirement.
3. **Retry multiplication is real billable spend (second pass, carried):** the solver permits up to three provider attempts per cell (`live_model_solver.py:41,199-206,646-676`), and timeouts or 429/5xx responses may bill for inference that produced no usable result; workflow-level reruns multiply again. Required (bead 1e): a **max-billable-attempts policy** recorded in the execution policy — per-cell attempt budget, attempt identity persisted with results (a **provider-call attempt journal** whose durable unit is the **individual HTTP attempt**, not the logical call — keyless plan §4e′ rev 7 defines the row contract: prompt digest, attempt identity, pre-call reservation, raw + normalized response, transactional settlement), so reruns resume instead of re-billing. **Rev-4 correction (fourth pass, accepted): "halt a shard after k consecutive cell failures" is not implementable** — the eight shards run as independent matrix jobs under `fail-fast: false` in parallel (`run-benchmark.yaml:875-887`); there is no shared ordering in which "consecutive" means anything, and more calls are already in flight when any failure is observed. Replace with a **shared atomic circuit breaker**: a durable failure counter (same store as the reservation ledger) that every job checks-and-increments *immediately before each provider call*; when failures within a sliding window cross the threshold, the breaker trips and all subsequent pre-call checks refuse, bounding further spend to at most one in-flight call per job.
