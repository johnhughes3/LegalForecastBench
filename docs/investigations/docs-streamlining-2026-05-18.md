#### Repetition clusters and canonical source-of-truth assignments

1. **Root landing page vs docs index.** Root `README.md` already has a public status/quickstart/docs list (`README.md:3-11`, `README.md:36-49`, `README.md:101-113`), while `docs/README.md` is the fuller navigation index (`docs/README.md:7-42`). **Recommendation:** keep `README.md` as the public landing page and truth-in-advertising status page; make `docs/README.md` the only full documentation index. Consolidate the root docs list to a short pointer plus only the highest-value links. Do not remove tested root phrases without updating doc-contract tests (`tests/test_public_alpha_docs.py:8-17`).
2. **Pre-data alpha / no official leaderboard language.** This status repeats in the root landing page (`README.md:3-11`, `README.md:24-34`), docs index (`docs/README.md:44-46`), methodology (`docs/methodology.md:3-6`), preregistration (`docs/preregistration.md:11-14`), acquisition (`docs/acquisition.md:3-6`, `docs/acquisition.md:91-96`), and release notes (`docs/v0.1_alpha_release_notes.md:7-9`). **Recommendation:** canonicalize current public status in `README.md`; in topic docs keep only topic-specific consequences (e.g., methodology gate, preregistration non-cycle warning, acquisition blocker) because tests pin several of those exact phrases (`tests/test_public_alpha_docs.py:36-44`; `tests/test_production_acquisition_docs.py:8-16`).
3. **Result-tier and canonical-leaderboard rules.** Root README summarizes all tiers (`README.md:139-150`), methodology repeats official/community separation (`docs/methodology.md:258-271`), run-card template repeats publication caveats (`docs/run_card_template.md:87-98`), and `docs/result_tiers.md` is the full policy (`docs/result_tiers.md:7-27`, `docs/result_tiers.md:74-93`, `docs/result_tiers.md:116-157`). **Recommendation:** make `docs/result_tiers.md` canonical; reduce root/methodology/run-card prose to pointers plus the tested phrases `Result tiers`, `docs/result_tiers.md`, and `Only official results are canonical` (`tests/test_result_tier_docs.py:26-35`; `tests/test_result_tier_policy_docs.py:8-31`).
4. **Acquisition blocker and fallback path.** Root README gives the high-level blocker (`README.md:32-34`), `docs/acquisition.md` gives operator commands and guardrails (`docs/acquisition.md:31-89`), methodology gives the detailed Phase 0 protocol revision (`docs/methodology.md:48-75`), data card gives cycle-report wording (`docs/data_card.md:38-43`), and preregistration gives official-readiness criteria (`docs/preregistration.md:55-71`). **Recommendation:** keep `docs/acquisition.md` canonical for CLI/operator instructions and live/paid guardrails; keep methodology canonical for why the protocol changed; keep preregistration for readiness gates only. Tests require current acquisition command names and flags, and require stale flags to stay absent (`tests/test_production_acquisition_cli_docs.py:8-48`).
5. **Benchmark task, validity standard, and non-representativeness.** Root README gives the elevator pitch (`README.md:3-6`), methodology gives the task/unit/metric and validity standard (`docs/methodology.md:8-23`, `docs/methodology.md:77-99`), data card repeats intended dataset and no population-rate claim (`docs/data_card.md:7-17`), and ethics repeats misuse/non-representativeness language (`docs/ethics.md:3-5`, `docs/ethics.md:86-93`). **Recommendation:** make `docs/methodology.md` canonical for task, unit, metric, and validity standard; keep data card cycle-specific; keep ethics misuse-focused.
6. **Outcome leakage / contamination / legitimate signal.** Methodology is already the most complete source (`docs/methodology.md:101-150`, `docs/methodology.md:152-158`, `docs/methodology.md:203-234`). Data card lists inclusion/exclusion consequences (`docs/data_card.md:45-72`, `docs/data_card.md:96-106`), ethics gives the risk framing (`docs/ethics.md:40-47`), and preregistration says what to lock before scoring (`docs/preregistration.md:28-53`, `docs/preregistration.md:92-109`). **Recommendation:** make methodology canonical; shorten data card/ethics/preregistration to their operational fields and cross-reference methodology for definitions.
7. **Public/private boundary, sensitive materials, redistribution, and takedown.** Boundary language repeats across data card (`docs/data_card.md:108-119`, `docs/data_card.md:140-169`), ethics (`docs/ethics.md:27-38`, `docs/ethics.md:95-106`), private storage layout (`docs/private_storage_layout.md:11-33`, `docs/private_storage_layout.md:57-110`, `docs/private_storage_layout.md:145-160`), private store export (`docs/private_store_export.md:61-71`), official aggregation (`docs/official_aggregation.md:65-72`), official eval environment (`docs/official_eval_environment.md:62-68`), and withdrawal workflow (`docs/withdrawal_workflow.md:8-17`, `docs/withdrawal_workflow.md:53-56`, `docs/withdrawal_workflow.md:93-105`). **Recommendation:** split canonical ownership by layer: ethics/data card for policy and cycle disclosure; private storage layout for prefixes/object classifications/manifest fields; private store export for export command and staging; official aggregation for final `public/` vs `private-debug/`; withdrawal workflow for takedown ledger and future-run exclusion. Avoid repeating full forbidden-material lists except where tests pin them (`tests/test_private_storage_layout_docs.py:47-66`; `tests/test_private_store_export_docs.py:36-48`; `tests/test_withdrawal_workflow_docs.py:59-69`; `tests/test_official_aggregation_docs.py:33-40`).
8. **Official evaluation environment vs storage layout.** `docs/official_eval_environment.md` is the full protected GitHub/OIDC/workflow source (`docs/official_eval_environment.md:3-27`, `docs/official_eval_environment.md:28-78`, `docs/official_eval_environment.md:91-106`), while storage layout repeats packet-read boundaries (`docs/private_storage_layout.md:129-143`) and withdrawal repeats that packet-read roles cannot read withdrawn/quarantine (`docs/withdrawal_workflow.md:11-17`, `docs/withdrawal_workflow.md:97-101`). **Recommendation:** keep official eval canonical for workflow security; keep storage layout as access matrix; shorten withdrawal to withdrawal-specific consequences.

10. **Preregistration, freeze order, and model registry fields.** `docs/preregistration.md` already owns required artifacts, human-entered fields, generated hashes, freeze order, model registry, and template locations (`docs/preregistration.md:16-27`, `docs/preregistration.md:28-53`, `docs/preregistration.md:73-90`, `docs/preregistration.md:92-140`). `docs/preregistration_template.md` is the human draft template (`docs/preregistration_template.md:3-24`, `docs/preregistration_template.md:25-65`, `docs/preregistration_template.md:91-121`). **Recommendation:** keep both, but make the relationship explicit: `preregistration.md` is policy/process; `preregistration_template.md` is a fillable checklist; `protocols/cycle_template.preregistration.yaml` remains machine-readable.
11. **Card/template overlap.** `docs/run_card_template.md` is a machine-readable JSON example/companion to `docs/run_card_schema.json` (`docs/run_card_template.md:3-11`, `docs/run_card_template.md:13-85`); `docs/model_card_template.md` is the human-readable companion pulling values from the model registry, run card, accounting, and leaderboard (`docs/model_card_template.md:3-5`, `docs/model_card_template.md:7-117`); `docs/data_card.md` is a cycle-level dataset/reconstruction artifact (`docs/data_card.md:3-5`, `docs/data_card.md:19-36`, `docs/data_card.md:140-154`). **Recommendation:** do not merge them.
