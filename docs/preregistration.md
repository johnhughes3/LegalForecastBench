# Preregistration Protocol

LegalForecast-MTD cycles should be preregistered before scored model runs. The
public record should live in OSF Registrations or AsPredicted, with a repo-local
copy in `protocols/<cycle_id>.preregistration.yaml`.

Preregistration is a guard against changing the benchmark after seeing labels or
model outputs. It also makes clear whether a run supports a pilot, rapid,
official descriptive, strong ranking, or annual aggregate claim.

The v0.1 public-feedback alpha is not an official preregistered cycle. It
publishes fixture-backed workflow evidence and release-governance docs so the
project can be reviewed before live packets are retained. Do not treat alpha
fixture outputs as preregistered benchmark results.

## Required Artifacts

Use three linked artifacts for each cycle:

- public registration: the OSF or AsPredicted timestamped record;
- protocol YAML: the machine-readable repo-local copy for validation;
- frozen hash bundle: hashes for the artifacts that must not drift before
  model evaluation.

The protocol YAML should be treated as the source used by code. The public
registration can summarize it and attach or reference the same fields.

## Human-Entered Fields

These fields are chosen before official collection and should be entered by the
benchmark operator:

- cycle ID and claim level;
- OSF or AsPredicted provider choice;
- model-release anchor and decision-date window;
- eligible courts and case sources;
- eligibility rules;
- exclusion rules;
- contamination filters;
- unitization rules;
- labeling and adjudication rules;
- primary metric and secondary metrics;
- bootstrap method and pairwise comparison plan;
- dominance and capping sensitivity rule;
- baseline definitions;
- case-mix diagnostic fields;
- model registry file path and intended model list;
- publication claim language.

For official cycles, these fields should be public or timestamped before model
evaluation starts. For rapid cycles, the lighter required set is the release
anchor, eligibility rule, exclusion rule, manifest hash, unit hash, scorer hash,
model registry hash, and a clear provisional-status label.

## Official-Cycle Readiness

The Phase 0 post-feasibility pilot blocks official evaluation until the live
retrieval path is proven. An official descriptive or ranking cycle should not be
preregistered for model execution until the fallback pilot records:

- at least 50 clean packets or a credible path to 50-100 clean packets;
- source-class counts for `case.dev-only`, `case.dev-plus-fallback`, and
  `excluded`;
- retained-packet case-mix diagnostics rather than search-hit diagnostics;
- linkage, leakage, text-quality, and document-completeness exclusion counts;
- separate case.dev discovery cost and fallback reconstruction cost totals;
- label ambiguity and lawyer review-time measurements for reviewed packets.

If the fallback pilot shows that most retained packets require reconstruction,
the preregistration should describe the source model as case.dev discovery plus
public-record fallback retrieval, not as case.dev-only packet construction.

## Generated Fields

These fields should be generated by code once artifacts exist:

- manifest SHA-256;
- prediction-unit SHA-256;
- label SHA-256;
- prompt SHA-256;
- scorer SHA-256;
- harness SHA-256 or version hash;
- model registry SHA-256;
- baseline-definition SHA-256;
- freeze timestamp;
- validation timestamp and validation result.

Generated fields should not be hand-edited after the freeze step. If a source
artifact changes, regenerate the hash bundle and record the reason before any
official model evaluation proceeds.

## Freeze Order

The clean sequence is:

1. Pre-register eligibility, exclusion, contamination, unitization, metrics,
   bootstrap, pairwise comparison, dominance, and publication-claim rules.
2. Build the candidate manifest.
3. Freeze and hash the manifest.
4. Freeze and hash prediction units.
5. Freeze and hash outcome labels.
6. Freeze and hash the prompt, scorer, harness version, model registry, and
   baseline definitions.
7. Validate the protocol YAML against all required fields and hashes.
8. Run model evaluations.
9. Publish results with the same protocol and hash bundle.

Labels do not need to be publicly released before model evaluation, but they
must be frozen and hashed before scoring starts.

## Model Registry

The model list is a first-class artifact. Preregistration, Inspect execution,
cost logging, run cards, model cards, and leaderboard reporting should all read
from the same model registry/run matrix.

Each registry entry should include:

- provider;
- model ID;
- display name;
- model version or snapshot;
- release timestamp when available;
- provider training cutoff status and cutoff date when known;
- sampling parameters;
- network and search policy;
- tool policy;
- context limit;
- pricing source and token prices;
- known cutoff or publicity caveats.

The registry hash is part of the freeze bundle. Changing the model registry
after seeing labels or outputs creates a new cycle or a documented amended run.

## Template

Use `docs/preregistration_template.md` for a human-readable draft and
`protocols/cycle_template.preregistration.yaml` for the machine-readable cycle
file. Fill both from the same choices, then rely on code validation to detect
missing hashes or stale field names before official evaluation.
